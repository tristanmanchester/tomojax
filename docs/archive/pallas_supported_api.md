# Supported Pallas API Contract

TomoJAX alignment is now lightning-first. The `lightning` alignment profile
requests Pallas for supported hot paths by default; `tortoise` keeps the older
JAX/FP32/reference posture. Pallas support is still explicit at each API surface:
the runtime support check must accept the geometry, detector grid, dtype, and
backend before a Pallas path can be reported as the actual backend.

## Backend Selection

Supported public backend values are:

- `jax`: reference implementation and gradient-safe fallback path.
- `pallas`: accelerator request. Unsupported or gradient-critical uses
  fall back to `jax` or raise a documented validation error at the Python/API
  boundary.

Alignment CLI usage:

```text
tomojax-align --align-profile tortoise
tomojax-align --align-profile lightning --projector-backend jax
tomojax-align --projector-backend pallas
```

Config files use the same key:

```toml
projector_backend = "pallas"
```

## Alignment-First Differentiability

Differentiability is a stage property, not a global lightning promise. When an
alignment stage needs derivatives with respect to pose or setup geometry, a
Pallas request uses JAX fallback unless the Pallas derivative path is explicitly
supported and tested for that surface. Proposal and scoring stages may be
performance-only when downstream refinement or verification accepts the result.

This means a Pallas request may still report:

- `requested_backend = "pallas"`
- `actual_backend = "jax"`
- `status = "fallback"`
- `eligible_for_speed_claim = false`

That is expected behavior, not a failed run, when differentiability is required.

## Current Support Matrix

| Surface | Status | Differentiability | Notes |
|---|---|---|---|
| General-pose forward projection | supported opt-in | JAX fallback for gradient-critical alignment use | Pallas parity is checked against JAX. |
| Forward residual SSE | supported opt-in for plain L2 performance scoring | performance-only unless a gradient-safe path is proven | Loss-adapter semantics remain owned by alignment objectives. |
| Alignment objective scoring | supported backend request | gradient-safe through JAX fallback | Backend provenance reports requested and actual backend. |
| FISTA core alignment reconstruction | supported opt-in for alignment reconstruction stages | performance-only unless used through existing differentiable reference tests | Public reconstruction defaults remain unchanged. |
| Specialized regular-z Pallas FBP helper | experimental/performance-only | not part of the alignment contract | Benchmark labels must not call this public `fbp()` timing unless public `fbp()` actually uses it. |

## Provenance

Workflow and benchmark metadata should report:

- requested backend
- actual backend
- support status
- fallback reason, when present
- API surface
- differentiability class
- speed-claim eligibility

Guard benchmark results remain optimization evidence. They are not publication
evidence unless generated by a publication-mode suite on the recorded machine
and environment.
